# 온프렘 코딩 어시스턴트 — 설계 문서

> 이 문서는 구현 전에 "무엇을, 왜, 어떻게" 만들지 정리한 설계 기록이다.
> 코드보다 먼저 읽히는 자리에 둔다. 빈 칸(트러블슈팅·효과)은 구현하면서 채운다.

---

## 1. 목표

코드·데이터가 외부로 나가면 안 되는 규제 산업(은행 등) 환경에서도 쓸 수 있도록,
**LLM 백엔드를 통째로 교체할 수 있는 코딩 어시스턴트**를 직접 설계·구현한다.

이 프로젝트가 증명하려는 것은 "또 하나의 챗봇"이 아니라 다음이다.

- 코딩 어시스턴트를 처음부터 설계하고 구현할 수 있다는 것
- 그 과정에서 내린 기술적 판단과 근거
- 막힌 지점을 어떻게 진단하고 해결했는가

핵심 가치 두 가지: **(1) 모델 교체 가능성**, **(2) 데이터가 보안 경계를 벗어나지 않는 구조**.
"데이터가 안 샌다"는 결과가 아니라, 그것을 가능하게 하는 설계를 보여주는 것이 목적이다.

---

## 2. 아키텍처 개요

클라이언트-서버 분리 구조. 백엔드는 **모듈러 모놀리스**(단일 배포, 내부는 모듈로 분리).
MSA가 아니다 — 내가 만들고 배포하는 서비스는 백엔드 하나이며,
Ollama·DB·외부 API는 그 백엔드가 갖다 쓰는 외부 의존성이다.

```
[ VSCode 확장 (TS) ]         ← 클라이언트(리모컨). 얇게 유지.
        │ HTTP / SSE
        ▼
┌─────────────────────────────────────────────┐
│  온프렘 서버 (사내망 = 보안 경계)              │
│                                               │
│   [ Spring 백엔드 (Spring AI) ]  ← 오케스트레이션 │
│        │                  │                   │
│        ▼                  ▼                   │
│   [ 컨텍스트 조회 ]   [ LLM 어댑터(ChatClient) ]─┼──▶ [ 외부 API ]  (경계 밖)
│        │                  │                   │      Claude / GPT
│        ▼                  ▼                   │
│   [ DB (MongoDB) ]    [ 로컬 모델(Ollama) ]    │
│        │                                       │
│        ▼                                       │
│   [ 감사 로그 ]                                 │
└─────────────────────────────────────────────┘
```

규제 모드에서는 LLM 어댑터가 외부 경로를 차단하면 어떤 데이터도 경계를 넘지 않는다.
강제 지점이 백엔드 한 곳에 모여 있다는 것이 분리의 핵심 명분이다.

## 3. 기술 선택 근거 (ADR)

각 결정은 `맥락 → 선택 → 근거 → 트레이드오프` 순으로 기록한다.

### ADR-1. 클라이언트와 백엔드를 분리한다

- **맥락**: 확장(TS) 하나에 LLM 호출까지 다 넣을 수도 있다.
- **선택**: 확장은 얇은 클라이언트로 두고, LLM 호출·정책·로그는 백엔드에 둔다.
- **근거**: "외부 모델 금지" 같은 정책의 **강제 지점이 한 곳이어야** 보안 경계가 의미를
  가진다. 정책을 확장에 두면 개발자 노트북 수백 대를 신뢰해야 하지만,
  백엔드에 두면 서버 한 대만 통제하면 된다. 라우팅·정책·로그가 한곳에 모인다.
- **트레이드오프**: 코드베이스 2개, 배포 대상 2개, 데모 시 둘 다 기동. 단순함을 일부 포기.

### ADR-2. 모델 호출은 Spring AI `ChatClient`로 추상화한다

- **맥락**: Ollama·Claude 모두 HTTP API이므로 `RestClient`로 직접 호출도 가능하다.
- **선택**: Spring AI의 `ChatClient` 추상화를 사용한다.
- **근거**: 프로바이더마다 요청/응답 형식이 다르다. 직접 호출하면 변환 코드를
  프로바이더별로 짜고 유지보수해야 한다. 이 프로젝트의 목적이 "교체 가능성"인데,
  그 교체를 검증된 추상화 위에서 보여주는 것이 설계 의도와 맞다.
  교체는 `application.yml`에서 모델만 바꾸면 되고 비즈니스 코드는 안 건드린다.
- **트레이드오프**: 프레임워크가 아직 지원 안 하는 최신 모델 기능은 제약될 수 있다.
  단, 채팅·스트리밍·도구 호출은 커버되므로 본 용도에 문제없음.

### ADR-3. 사내 데이터는 Phase 0에서 직접 조회, 이후 MCP로 승격한다

- **맥락**: 모델에 사내 컨텍스트(스키마, 유사 코드 등)를 물리는 방법은 두 가지다 —
  백엔드가 직접 DB를 조회해 프롬프트에 끼워넣기(push) vs 모델이 도구를 스스로
  호출해 가져가기(pull, MCP).
- **선택**: 초기에는 직접 조회로 단순하게 시작하고, 동작이 검증된 뒤 MCP로 전환한다.
- **근거**: 질문 유형이 다양한 코딩 어시스턴트는 "무엇이 필요한지 미리 모르는" 경우가
  많아 결국 pull(MCP)이 맞다. 다만 MCP 미경험 상태에서 처음부터 분리하면 디버깅 변수가
  많아져 완성 위험이 커진다. 직접 조회 → MCP 전환 과정 자체가 "기술 도입 판단"의
  흔적으로 남아 포트폴리오 가치가 높다. MCP는 표준이라 도구를 다른 AI 클라이언트도
  재사용할 수 있다는 장점도 있다.
- **트레이드오프**: 전환 단계가 생겨 작업량이 늘지만, 단계적 검증으로 디버깅이 쉬워진다.

### ADR-5. 연속 대화를 위해 ChatMemory를 Phase 0부터 켠다

- **맥락**: 코딩 어시스턴트는 "방금 답에 이어서" 식 후속 질문이 잦다.
  단발성으로 시작하면 요청 계약을 나중에 깨야 한다.
- **선택**: Phase 0부터 `MessageChatMemoryAdvisor` + `MongoChatMemoryRepository`로
  연속 대화를 켠다. 요청에 `conversationId`를 처음부터 포함한다.
- **근거**: Mongo를 이미 골라 저장소가 자동 설정(`ai_chat_memory` 컬렉션 자동 생성)되어
  추가 비용이 거의 없다. 비싼 건 advisor가 아니라 확장↔백엔드 요청 모양이라,
  그 계약을 먼저 굳히는 게 핵심. conversationId는 확장이 소유하고 "새 대화"가 리셋.
- **트레이드오프**: 윈도우(기본 20턴)를 넘는 오래된 맥락은 잘린다.
  선택 코드가 매 턴 쌓이면 토큰을 빨리 먹으므로 윈도우 크기로 관리.

### ADR-6. Spring Boot 4.0 + Spring AI 2.0을 선택한다

- **맥락**: 새 프로젝트에서 LTS 3.x로 시작하는 방법도 있다. 3.4는 2025년 EOL.
- **선택**: Boot 4.0 + Spring AI 2.0 (GA 출시 직후)을 기준으로 한다.
- **근거**: EOL이 임박한 3.x로 새 포트폴리오를 시작하는 것은 어색하다.
  AI 2.0은 GA 출시 직후라 공식 레퍼런스와 버전이 일치하고, ChatMemory·MCP 등
  핵심 기능이 안정 API로 확정되어 있다. 최신 버전 선택 자체가 결정의 흔적.
- **트레이드오프**: 출시 직후라 생태계 레퍼런스(블로그·StackOverflow)가 아직 얇다.
  공식 문서와 소스를 우선 참조해야 하는 구간이 있다.

### ADR-4. 감사 로그에 "외부 전송 여부" 플래그를 둔다

- **맥락**: 규제 환경에서 감사관이 가장 보고 싶은 것은 "경계 밖으로 나간 요청"이다.
- **선택**: (사용자, 시각, 모델종류, 외부전송여부, 프롬프트, 응답)을 기록한다.
  요청 시작과 완료/실패를 각각 남긴다.
- **근거**: 외부 호출 실패·타임아웃까지 추적해야 감사가 완전해진다.
- **트레이드오프 / 한계**: 프롬프트·응답 본문을 통째로 저장하면 **로그 DB 자체가
  민감 데이터 덩어리**가 되어 새 유출 구멍이 된다. 본문 보호(마스킹/암호화)는
  추후 과제로 남긴다 — 인지하고 있다는 것을 명시.

## 4. 데이터 흐름

1. 개발자가 코드 선택 + 질문 입력 → VSCode 확장
2. 확장이 (conversationId + 코드 + 질문)을 HTTP로 Spring 백엔드에 전송
3. 백엔드가 필요 시 사내 컨텍스트 조회 (Phase 0: 직접 DB 조회 / Phase 2+: MCP 도구 호출)
   - 방향: DB → 모델. 답 품질을 위한 **읽기**.
4. LLM 어댑터가 모델 호출 — 로컬(Ollama, 경계 안) 또는 외부 API(경계 밖)
   4-1. ChatMemory: 모델 호출 직전 advisor가 해당 conversationId의 직전 N턴을
        읽어 프롬프트에 주입(읽기), 호출 직후 새 턴을 저장(쓰기). `ai_chat_memory` 컬렉션.
5. 응답이 백엔드 → 확장 → 에디터에 표시
6. 프롬프트 + 응답을 감사 로그로 DB에 저장
   - 방향: 모델 → DB. 기록을 위한 **쓰기**. (3번의 컨텍스트 조회와 별개)

> 주의: DB 접점이 이제 셋이다 —
> ③ 컨텍스트(읽기, 답 품질) / ChatMemory(읽기+쓰기, 대화 연속성) / ⑥ 감사로그(쓰기, 규제 증거).
> 셋 다 Mongo지만 출처·목적·컬렉션이 다르다. 특히 ChatMemory와 감사로그를 한 컬렉션에 합치지 말 것
> — 전자는 모델이 다시 읽는 용도(자동 관리·TTL), 후자는 감사관용(본문 마스킹 한계가 붙는 ADR-4).

## 5. 모델 전략

| 구분 | 모델 | 용도 | 데이터 경계 |
|------|------|------|-------------|
| 외부 | Claude / GPT / GLM-5.2:cloud | 평소 개발, 빠른 데모 | 경계 밖 (추론이 외부 GPU) |
| 로컬 | `qwen2.5-coder:3b` (Ollama) | 온프렘 시연 데모 | 경계 안 (추론이 내 장비) |
| 프로덕션(참고) | `qwen3.6:35b` 등 | GPU 서버 확보 시 | 경계 안 |

- 외부 API는 추론을 위해 데이터가 외부로 나가야 하지만, Ollama는 모델을 로컬에 두고
  추론하므로 데이터가 경계를 벗어나지 않는다. 이것이 "데이터 안 샘"의 실제 메커니즘.
- 개발 환경: M1 / 16GB. 통합 메모리 + Metal 가속으로 `3b`는 무난, `7b`는 빠듯.
  `35b`(24GB)는 16GB에 올라가지 않으므로 데모 대상 아님 → README에 프로덕션 권장으로만 명시.
- 호출은 `ChatClient`에 시스템 프롬프트 + 유저 메시지(선택 코드 + 질문)를 넘기고
  스트리밍으로 수신. 응답 품질은 모델 성능 + **컨텍스트 주입 설계**가 좌우.

DB는 MongoDB로 확정 — ChatMemory 백엔드 자동 지원(`ai_chat_memory` 컬렉션 자동 생성) + 로그 스키마 유연성.

## 6. 구현 단계

- **Phase 0** — 동작하는 뼈대: 확장(명령 1개) + 백엔드 `POST /chat` + ChatClient
  + **ChatMemory(연속 대화, conversationId 포함)**. end-to-end + 후속 질문 1턴.
- **Phase 1** — 프로바이더 교체: `application.yml` 토글, Ollama 로컬 모델 연결.
- **Phase 2** — MCP: 직접 조회를 MCP 서버/도구로 승격.
- **Phase 3** — 컨텍스트 & 감사 로그: 코드 컨텍스트 수집, 로그 저장.
- **Phase 4** — 온프렘 패키징: docker-compose(백엔드 + Ollama + DB), 데모 영상/GIF.

## 9. 배포 / 증거 전략

- 확장: VS Code Marketplace에 publish (무료, publisher 이름이 URL에 박힘) → 증거.
- 백엔드 URL·LLM 키는 VSCode 설정값으로 외부화 (BYO-key). 하드코딩 금지.
  → self-host 설계 그 자체이자, 타인이 내 키로 비용 태우는 것 방지.
- 백엔드는 데모 시 로컬 기동, 작동 화면을 영상/GIF로 기록.
- 증거 묶음: 마켓 등록 URL + GitHub 레포(이 문서 + docker-compose + 데모 GIF) + 데모 영상.


## 10. 서비스 업데이트
- 확장: 그때 그때, 이 서비스를 고도화 시키기 위해 추가하면 좋은 기술이나 기능 추천.

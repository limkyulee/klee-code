# 로그인과 히스토리를 붙이며 다중 사용자 서비스로 바뀐 과정

관련 커밋: `69197b0 Session Logic add`, `3c4acd3 Chat History`, `27237b5 Conversation Collection Add - Conversation detail, delete`, `9285ba7 multi conversation`

초기 Klee Code는 사실상 단일 사용자 도구였다. VS Code 확장에서 질문을 보내고, 백엔드가 답을 돌려주는 구조였다. 하지만 온프레미스 환경에서 여러 사용자가 같은 백엔드를 쓴다고 생각하면 사용자 개념이 필요하다.

누가 질문했는지, 누구의 대화 히스토리인지, 어떤 설정을 쓰는지 구분해야 한다. 그래서 V2에서 로그인, 세션, 사용자 설정, 히스토리 기능을 붙였다.

## 세션을 붙이며 서비스의 형태가 바뀌었다

이 단계에서 백엔드에는 `auth`, `user`, `security` 도메인이 추가됐다. 사용자는 `userId + password`로 가입하고, 비밀번호는 BCrypt로 해시해서 저장했다.

세션은 access token과 refresh token으로 나눴다.

- access token은 JWT로 발급하고 API 호출 시 `Authorization: Bearer` 헤더로 보낸다.
- refresh token은 opaque token으로 만들고, MongoDB에는 SHA-256 해시만 저장한다.
- refresh session에는 TTL을 둬 만료되게 한다.
- VS Code 확장은 refresh token을 SecretStorage에 저장하고 access token은 메모리에 둔다.

이 구조를 붙이면서 Klee Code는 "내 로컬에서 실행하는 확장"에서 "사용자 단위로 상태를 가지는 서비스"로 바뀌었다.

## 설정 화면을 전면에 두지 않았다

로그인과 설정 기능을 붙이다 보면 앱의 첫 화면이 로그인 폼이 되기 쉽다. 하지만 Klee Code의 첫 화면은 채팅이어야 한다고 봤다.

그래서 초기 로그인 화면과 설정 화면을 먼저 띄우는 방식은 피했다. 대신 헤더 우측 아이콘에서 로그인, 회원가입, 로그아웃, 설정, 히스토리, 새 대화를 열 수 있게 했다.

이 판단은 UI 취향 문제가 아니었다. 사용자가 이 확장을 여는 이유는 설정을 하기 위해서가 아니라 질문하기 위해서다. 인증과 설정은 필요하지만, 제품의 중심은 채팅이어야 했다.

## 히스토리는 audit log 목록이 아니었다

처음에는 `audit_logs`를 조회하면 히스토리도 만들 수 있을 것 같았다. 감사 로그에는 질문과 답변이 들어 있으니, 이를 목록으로 보여주면 된다고 생각할 수 있다.

하지만 실제 채팅 UI에서 사용자가 원하는 히스토리는 턴 목록이 아니다. 한 대화창에서 여러 번 질문했다면 히스토리에는 하나의 대화로 보여야 한다.

그래서 먼저 `/audit/chat-history`를 `conversationId` 단위 요약 목록으로 재설계했다. 개별 로그가 아니라 대화 단위로 묶어 보여주기 위한 작업이었다.

이후에는 `conversations` 컬렉션을 추가했다. 이 컬렉션은 대화 메타데이터를 담당한다. 제목, 마지막 메시지, 업데이트 시각, 소유 사용자 같은 목록 표시용 정보를 가진다.

저장 책임은 이렇게 나눴다.

```text
conversations  -> 대화 목록과 메타데이터
audit_logs     -> 턴 단위 질문/답변과 감사 기록
ai_chat_memory -> 모델이 이어서 답하기 위한 메모리
```

이 구분이 생기면서 대화 목록, 상세 복원, 이어가기, 완전 삭제가 자연스럽게 구현됐다.

## 대화 복원과 삭제까지 가야 히스토리다

히스토리는 목록만 보여주면 끝이 아니다. 이전 대화를 눌렀을 때 화면에 메시지가 복원되어야 하고, 같은 `conversationId`로 이어서 질문할 수 있어야 한다. 필요하면 삭제도 되어야 한다.

그래서 conversation 기반 API와 Webview UI를 함께 바꿨다. 기존 `/audit/chat-history`는 바로 제거하지 않고 새 conversation 목록으로 위임해 호환성을 유지했다.

이미 쌓인 감사 로그와 새 conversation 구조 사이에는 lazy backfill 개념도 들어갔다. 모든 과거 데이터를 한 번에 마이그레이션하지 않고, 필요한 시점에 conversation 메타데이터를 보완하는 방식이다. 초기 프로젝트 단계에서는 이 방식이 더 현실적이었다.

## 다중 대화를 다루면서 생긴 변화

이후 multi conversation 작업을 거치면서 Webview는 단순히 현재 메시지 배열만 들고 있는 화면이 아니게 됐다. 사용자는 여러 대화를 오가고, 이전 대화를 열고, 새 대화를 시작한다.

이 시점에서 Phase 0부터 `conversationId`를 넣어둔 선택이 다시 의미가 생겼다. 처음에는 OutputChannel에 답을 찍는 정도였지만, 대화 ID라는 계약을 유지했기 때문에 대화 복원과 이어가기를 같은 축에서 구현할 수 있었다.

로그인과 히스토리를 붙인 뒤 Klee Code는 확실히 다중 사용자 서비스에 가까워졌다. 그리고 저장소도 목적별로 나뉘었다. 이 변화가 다음 단계인 LLM Gateway 리팩토링의 기반이 됐다.

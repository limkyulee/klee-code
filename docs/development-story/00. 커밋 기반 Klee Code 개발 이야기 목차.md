# 커밋 기반 Klee Code 개발 이야기

이 원고는 `git log`를 기준으로 Klee Code가 어떻게 바뀌었는지 다시 정리한 개발 기록이다. 처음부터 지금 같은 채팅 UI와 로그인, 히스토리, 중앙 LLM Gateway가 있었던 것은 아니었다.

초기에는 VS Code 명령 하나와 백엔드 `/chat` API로 agent 대화가 되는지만 확인했다. 답변도 별도 채팅 UI가 아니라 VS Code OutputChannel에 찍었다. 이후 Webview 패널을 붙였고, HTML 문자열로 만든 UI를 React로 옮겼고, SSE 스트리밍과 Markdown 렌더링을 다듬었다. 그 다음에 코드 컨텍스트, 감사 로그, 로그인, 세션, 대화 히스토리, 온프레미스 LLM Gateway가 차례대로 들어왔다.

정리 기준은 기능 목록이 아니라 시간순 변화다. 각 글은 관련 커밋을 함께 적어두었다.

1. [OutputChannel에서 시작한 Phase 0](./01.%20OutputChannel에서%20시작한%20Phase%200.md)
2. [채팅 UI가 생기고 React Webview로 갈아탄 과정](./02.%20채팅%20UI가%20생기고%20React%20Webview로%20갈아탄%20과정.md)
3. [SSE 스트리밍과 답변 렌더링을 다듬은 과정](./03.%20SSE%20스트리밍과%20답변%20렌더링을%20다듬은%20과정.md)
4. [코드 컨텍스트와 감사 로그가 들어오면서 생긴 설계 분리](./04.%20코드%20컨텍스트와%20감사%20로그가%20들어오면서%20생긴%20설계%20분리.md)
5. [로그인과 히스토리를 붙이며 다중 사용자 서비스로 바뀐 과정](./05.%20로그인과%20히스토리를%20붙이며%20다중%20사용자%20서비스로%20바뀐%20과정.md)
6. [개인 모델 설정에서 중앙 LLM Gateway로 돌아선 이유](./06.%20개인%20모델%20설정에서%20중앙%20LLM%20Gateway로%20돌아선%20이유.md)
7. [실행 가능한 포트폴리오로 만들기 위해 정리한 것들](./07.%20실행%20가능한%20포트폴리오로%20만들기%20위해%20정리한%20것들.md)

블로그에 옮길 때는 1~7편으로 나눠도 되고, 하나의 긴 회고 글로 합쳐도 된다. 포인트는 “처음부터 완성된 설계가 아니라, 문제를 확인하면서 설계를 계속 좁혀간 프로젝트”라는 흐름이다.

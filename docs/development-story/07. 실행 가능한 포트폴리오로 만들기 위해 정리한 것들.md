# 실행 가능한 포트폴리오로 만들기 위해 정리한 것들

관련 커밋: `467a050 Dockerfile setting`, `e818679 fix md file path`, `cb5636a fix extension compile error`, `704b73f fix icon`

기능이 늘어나면서 마지막에 계속 잡아야 했던 것은 실행 환경이었다. 포트폴리오 프로젝트는 코드만 있어서는 부족하다. 다른 사람이 봤을 때 구조가 이해되어야 하고, 실행 경로도 어느 정도 재현 가능해야 한다.

Klee Code는 VS Code Extension, Spring Boot Backend, MongoDB, Ollama가 함께 움직인다. 이 네 가지가 맞물리기 때문에 작은 설정 차이도 바로 실행 실패로 이어진다.

## Dockerfile과 Compose를 정리했다

Phase 4에서 백엔드 Dockerfile과 Docker Compose 구성을 추가했다. 목표는 거창한 운영 환경이 아니라, 온프레미스 구성을 설명할 수 있는 최소 실행 묶음을 만드는 것이었다.

Compose에서는 백엔드, MongoDB, Ollama, 모델 pull 작업을 같은 네트워크에 둔다. 이 환경에서는 백엔드가 `http://ollama:11434`로 Ollama를 찾을 수 있다.

반대로 로컬 개발에서는 백엔드를 호스트에서 실행할 수도 있다. 이때는 `ollama`가 아니라 `localhost`를 봐야 한다. 그래서 실행 방식에 따라 기본 URL을 다르게 봐야 한다는 기준이 생겼다.

이 차이를 모르면 "로컬에서는 ollama list가 되는데 백엔드에서는 모델이 안 보이는" 상황이 생긴다.

## start.sh는 단순 실행 스크립트가 아니었다

`start.sh`도 여러 번 수정됐다. 처음에는 백엔드 JAR 선택 문제가 있었다. Gradle 빌드 결과에는 실행 가능한 Spring Boot JAR와 plain JAR가 같이 생길 수 있는데, plain JAR를 실행하면 Manifest 오류가 난다.

그래서 스크립트가 실행 가능한 Boot JAR를 선택하도록 고쳤다.

이후에는 extension 빌드 로그도 개선했다. Node 버전이 맞지 않거나 Vite 빌드가 실패하면 사용자는 "확장이 안 뜬다"는 결과만 보게 된다. 그래서 `logs/extension-build.log`를 확인할 수 있게 하고, Node 버전 진단도 명확히 했다.

실행 스크립트는 편의 기능처럼 보이지만, 실제로는 문제를 재현하고 설명하기 위한 도구였다.

## 문서 위치도 정리했다

프로젝트가 커지면서 설계 문서가 루트와 `docs/`에 흩어졌다. `e818679` 커밋에서는 `DESIGN.md`, `DESIGN_V2.md`, `STRUCTURE_V2.md`, `TECH_STACK.md`, ERD 파일 등을 `docs/` 아래로 모았다.

이 작업은 기능 개발은 아니지만 중요했다. 포트폴리오에서 문서는 코드만큼 큰 증거다. 어떤 설계를 했는지, 왜 그렇게 바꿨는지, 현재 구조가 무엇인지 한 곳에서 읽을 수 있어야 한다.

문서 위치를 정리하면서 시스템 아키텍처 이미지와 MongoDB ERD도 `docs/`에 들어갔다. 이후 블로그 원고를 작성할 때도 이 문서들이 기준점이 됐다.

## VS Code 실행 설정을 재현 가능하게 봤다

마지막에는 VS Code extension 실행 설정 문제도 있었다. `Could not find the task 'extension: compile'.` 오류가 발생했는데, 원인은 단순히 task label 하나가 틀린 수준을 넘었다.

루트 `.vscode/tasks.json`은 로컬에는 있었지만 Git 추적 대상이 아니고 `.gitignore`에 의해 ignored 상태였다. 즉 내 환경에서는 있는 것처럼 보여도, 다른 체크아웃 환경에서는 VS Code가 preLaunchTask를 찾지 못할 수 있었다.

이 문제를 겪고 나서 개발 실행 설정도 소스의 일부처럼 봐야 한다는 기준이 생겼다. 포트폴리오 프로젝트라면 "내 컴퓨터에서만 되는 설정"에 의존하면 안 된다.

VS Code 확장 호환 버전도 같은 맥락이었다. 최신 버전을 무조건 요구하는 것이 아니라, 실제 사용하는 API와 지원하려는 사용자 버전에 맞춰야 한다.

## 아이콘과 세부 UI도 마지막에 다듬었다

마지막 커밋에는 아이콘 관련 스타일 수정도 있었다. 기능적으로 큰 변화는 아니지만, 확장 패널이 실제 제품처럼 보이려면 이런 작은 부분도 정리해야 한다.

초기에는 OutputChannel에 답변만 찍던 프로젝트였다. 마지막에는 채팅 패널, 로그인, 히스토리, 중앙 LLM Gateway, Docker Compose, 문서와 아키텍처 이미지까지 갖춘 형태가 됐다.

이 과정을 보면 Klee Code는 한 번에 완성된 프로젝트가 아니다. 작은 end-to-end 검증에서 시작해, 문제를 만날 때마다 구조를 나누고, 실행 환경을 정리하면서 조금씩 제품 형태로 가까워졌다.

포트폴리오에서 보여주고 싶은 것도 이 지점이다. 처음부터 거대한 설계를 완성했다는 이야기가 아니라, 실제로 만들면서 잘못된 방향은 줄이고 필요한 구조는 남긴 과정이다.

# OutputChannel에서 시작한 Phase 0

관련 커밋: `52160bd Initial commit cheer up`, `07202e5 build package`, `2b3df19 phase 0`

Klee Code의 첫 모습은 지금과 많이 달랐다. 채팅 UI도 없었고, 로그인도 없었고, 대화 히스토리도 없었다. 처음 목표는 단순했다.

VS Code에서 코드를 선택하고 질문을 입력하면, Spring 백엔드가 LLM에 물어보고 답을 받아올 수 있는가.

그래서 Phase 0은 제품을 만드는 단계라기보다 연결을 검증하는 단계였다. VS Code 확장은 커맨드 하나를 등록했다. 사용자가 `Klee Code: Ask Assistant`를 실행하면 `showInputBox`로 질문을 받고, 선택된 코드를 함께 백엔드 `/chat` API로 보냈다. 응답은 별도의 채팅 화면이 아니라 VS Code `OutputChannel`에 `[Assistant]` 형태로 찍었다.

이 구조는 UI로 보면 투박하지만, 초기 검증에는 오히려 맞았다. React, Webview, CSS, 상태 관리 같은 변수를 빼고 가장 중요한 질문만 확인할 수 있었기 때문이다.

## 처음 검증한 것은 대화가 아니라 경계였다

초기 README와 설계 문서에는 이미 방향이 잡혀 있었다. Klee Code는 단순 챗봇이 아니라 규제 산업이나 내부망 환경에서 쓸 수 있는 코딩 어시스턴트로 시작했다. 핵심은 두 가지였다.

- VS Code 확장은 얇은 클라이언트로 둔다.
- LLM 호출과 정책 강제는 백엔드에 둔다.

이 판단이 먼저 있었기 때문에 Phase 0에서도 확장이 직접 모델을 호출하지 않았다. 확장은 선택 코드와 질문을 전달하는 역할만 했다. 실제 LLM 호출은 Spring 백엔드의 `ChatService`가 맡았다.

당시 `ChatService`는 Spring AI `ChatClient`를 주입받아 사용했다. 코드가 있으면 Markdown 코드 블록으로 감싸고, 질문과 함께 user message를 만들었다. 그리고 `conversationId`를 advisor parameter로 넘겼다.

중요한 점은 Phase 0부터 `conversationId`가 있었다는 것이다. 단발성 질문만 처리했다면 나중에 연속 대화를 붙일 때 요청 계약이 바뀌었을 것이다. 하지만 처음부터 `MessageChatMemoryAdvisor`와 MongoDB ChatMemory를 고려해 `conversationId`를 포함했다. 덕분에 나중에 UI와 히스토리가 붙어도 대화 단위라는 개념은 계속 이어갈 수 있었다.

## 왜 OutputChannel로 시작했나

처음부터 예쁜 채팅 UI를 만들 수도 있었다. 하지만 그 단계에서는 중요한 문제가 아니었다.

초기에는 다음이 먼저였다.

- VS Code 확장 명령이 정상 등록되는가.
- 선택 코드와 질문을 백엔드로 보낼 수 있는가.
- Spring AI `ChatClient`로 모델 응답을 받을 수 있는가.
- `conversationId`로 연속 대화의 기반을 만들 수 있는가.
- 백엔드 URL 같은 설정을 확장에서 읽을 수 있는가.

OutputChannel은 이 검증에 충분했다. 사용자 경험은 부족했지만, end-to-end 흐름을 가장 빨리 확인할 수 있었다.

이때 만든 구조는 단순했지만 이후 개발의 기준이 됐다. 확장은 리모컨처럼 얇게 두고, 백엔드가 LLM 호출과 정책을 책임진다는 방향이다.

## Phase 0에서 남긴 기준

Phase 0을 지나며 확실해진 것은 하나였다. 이 프로젝트는 “VS Code 안에 채팅창을 하나 붙이는 것”이 아니라, 백엔드가 중심이 되는 AI Coding Assistant여야 했다.

그래서 초기 구현은 작게 시작했다.

```text
VS Code Command
  -> showInputBox
  -> selected code + question
  -> POST /chat
  -> Spring AI ChatClient
  -> OutputChannel
```

이 구조로 먼저 대화가 되는지 확인했고, 그 다음에야 UI를 올렸다. 지금 보면 당연한 순서처럼 보이지만, 이 단계를 분리한 것이 좋았다. 모델 호출, 요청 계약, conversationId, 백엔드 설정을 먼저 잡아두었기 때문에 이후 Webview UI가 붙어도 핵심 흐름은 크게 흔들리지 않았다.

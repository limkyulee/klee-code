# 코드 컨텍스트와 감사 로그가 들어오면서 생긴 설계 분리

관련 커밋: `685a604 MongoDB Chat Audit Logging`

채팅 UI와 스트리밍이 어느 정도 잡히고 나니, 다음 문제는 답변의 품질과 추적성이었다.

사용자가 "이 코드 설명해줘"라고 물었을 때 모델이 제대로 답하려면 질문만으로는 부족하다. 어떤 파일의 어떤 언어인지, 선택한 범위가 어디인지, 주변 코드가 무엇인지 알아야 한다.

그래서 VS Code 확장에서 코드 컨텍스트를 수집하기 시작했다. 선택 코드, 파일 경로, 언어 ID, 선택 범위, 주변 라인 스니펫을 `ChatRequest`에 담아 백엔드로 보냈다.

이 데이터는 백엔드에서 다시 만들기 어렵다. 사용자가 현재 보고 있는 에디터 상태는 VS Code 확장이 가장 정확히 알고 있다. 그래서 컨텍스트 수집 책임은 extension host에 두고, 백엔드는 받은 컨텍스트를 프롬프트 구성과 감사 로그에 활용했다.

## 답변 품질을 위한 컨텍스트

코딩 어시스턴트에서 컨텍스트는 답변 품질의 핵심이다.

질문만 보내면 모델은 일반론으로 답할 가능성이 높다. 반대로 선택 코드와 주변 코드가 함께 들어가면 훨씬 구체적으로 답할 수 있다. 예를 들어 "여기서 예외 처리가 이상한가?"라는 질문은 코드 없이 보면 의미가 거의 없다. 하지만 파일 정보와 선택 범위가 있으면 실제 맥락을 기준으로 설명할 수 있다.

그래서 Phase 3에서는 `CodeContext`, `SelectionRange` 같은 DTO가 추가됐다. 단순히 `code` 문자열 하나만 보내던 요청이, 편집기 상태를 포함하는 요청으로 확장된 것이다.

## ChatMemory와 Audit Log는 다르다

이 시기에 저장소 책임도 다시 정리했다.

이미 Spring AI `MessageChatMemoryAdvisor`가 `ai_chat_memory` 컬렉션을 사용하고 있었다. 이 저장소는 모델이 연속 대화를 하기 위해 다시 읽는 데이터다. 같은 `conversationId`로 질문하면 이전 대화를 프롬프트에 주입한다.

하지만 감사 로그는 목적이 다르다. 감사 로그는 모델이 다시 읽기 위한 데이터가 아니다. 사람이 나중에 추적하기 위한 기록이다. 누가 어떤 질문을 했고, 어떤 코드 컨텍스트가 포함됐고, 어떤 모델로 요청했고, 성공했는지 실패했는지를 남기는 용도다.

그래서 `ai_chat_memory`와 `audit_logs`는 분리했다.

```text
ai_chat_memory -> 모델의 연속 대화를 위한 메모리
audit_logs     -> 사람과 운영자를 위한 요청/응답 추적 기록
```

같은 대화 데이터를 다루지만 읽는 주체가 다르다. 이 차이를 구분한 것이 이후 히스토리 설계에도 영향을 줬다.

## audit 도메인을 chat 하위에서 분리했다

초기 구현에서는 감사 로그가 채팅 흐름에 붙어 있었다. 기능을 빠르게 붙이기에는 괜찮지만, 도메인 경계가 애매했다.

채팅 도메인은 질문을 받고 LLM을 호출하는 책임을 가진다. 감사 로그는 요청과 응답의 흔적을 남기는 책임을 가진다. 두 기능은 함께 동작하지만 같은 도메인은 아니라고 봤다.

그래서 `audit`를 `chat` 하위가 아니라 별도 package root로 분리했다. 구조도 `controller`, `service`, `repository`, `dto` 기준으로 맞췄다. DTO와 단순 문서 객체는 가능하면 `record`로 바꿨다.

이 결정은 코드 스타일을 맞추기 위한 작업만은 아니었다. 감사 로그 저장이 실패하더라도 채팅 응답이 깨지면 안 된다. 그래서 `AuditLogService`가 저장 실패를 격리하고 경고 로그만 남기도록 했다. 채팅과 감사는 연결되어 있지만, 실패 전파는 분리해야 했다.

## 외부 전송 여부를 남긴 이유

초기 설계부터 Klee Code는 보안 경계를 중요하게 봤다. 그래서 감사 로그에는 모델 provider와 외부 전송 여부를 남기도록 했다.

로컬 Ollama를 쓰는 경우 `externalTransfer=false`로 기록된다. 이 값 하나로 보안이 완성되는 것은 아니지만, 설계 의도를 데이터로 남기는 데 의미가 있다. 나중에 특정 요청이 어떤 경로로 처리됐는지 설명할 수 있어야 하기 때문이다.

이 시점부터 Klee Code는 단순 채팅 도구에서 운영 관점의 흔적을 남기는 서비스로 바뀌었다. 코드 컨텍스트는 답변 품질을 위해 들어왔고, 감사 로그는 운영과 추적성을 위해 들어왔다. 둘을 같은 문제로 보지 않은 것이 이 단계의 핵심이었다.

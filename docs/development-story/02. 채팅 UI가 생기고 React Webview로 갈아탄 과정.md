# 채팅 UI가 생기고 React Webview로 갈아탄 과정

관련 커밋: `b14b8f6 Chat UI Build`, `1b4b7bd Chat UI Build`, `8bc1dea Html to React and Node version up`, `439a12a fix folder struct`

Phase 0에서는 응답을 OutputChannel에 찍었다. 개발 검증에는 충분했지만, 실제 코딩 어시스턴트라고 부르기에는 부족했다. 사용자는 질문과 답변을 대화처럼 보고 싶어 한다. 이전 질문도 화면에 남아 있어야 하고, 새 대화를 시작할 수 있어야 한다.

그래서 다음 단계에서 VS Code Webview 기반 채팅 패널을 만들었다.

처음 Webview는 React가 아니었다. `ChatViewProvider` 안에서 HTML 문자열을 직접 만들고, 그 안에 CSS와 스크립트를 넣었다. Extension Host가 WebviewView를 만들고, Webview에서 `ask`, `newConversation`, `ready` 같은 메시지를 보내면 Extension Host가 백엔드로 요청을 보내는 방식이었다.

이 단계에서 OutputChannel 중심 구조는 채팅 패널 중심 구조로 바뀌었다.

```text
VS Code Webview
  -> postMessage
  -> Extension Host
  -> Backend /chat
  -> Extension Host
  -> Webview message
```

## HTML 문자열 방식의 한계

HTML 문자열 방식은 빠르게 만들 수 있다는 장점이 있었다. 별도 프론트엔드 빌드가 필요 없고, 하나의 TypeScript 파일에서 Webview 화면까지 다 볼 수 있었다.

하지만 화면이 조금만 커져도 한계가 바로 보였다. 메시지 목록, 입력창, 상태 표시, 새 대화 버튼, 에러 메시지 스타일이 한 파일 안에 섞였다. UI 상태가 많아질수록 문자열 안의 스크립트도 커졌다.

이 구조로는 채팅 UI를 계속 키우기 어렵다고 판단했다.

그래서 Webview UI를 React로 옮겼다. Vite 설정을 추가하고, `webview-ui`를 별도 영역으로 분리했다. Extension Host는 VS Code API와 백엔드 통신을 담당하고, React Webview는 화면 상태와 렌더링을 담당하게 했다.

## React로 옮기면서 경계를 다시 나눴다

React로 바꾸면서 단순히 UI 라이브러리만 바뀐 것은 아니었다. 런타임 경계를 다시 잡았다.

Extension Host는 VS Code API를 사용할 수 있다. 활성 에디터에서 선택 코드를 읽고, SecretStorage에 접근하고, 백엔드 API를 호출할 수 있다. 반대로 Webview UI는 브라우저 환경에 가깝다. VS Code API를 직접 쓰지 않고 `acquireVsCodeApi`를 통해 메시지만 보낸다.

그래서 구조를 다음처럼 나눴다.

```text
extension/src/extension-host
  -> VS Code API
  -> backend API
  -> conversationId 관리
  -> Webview message handler

extension/webview-ui
  -> React 화면
  -> message reducer
  -> ChatInput / MessageList / MessageBubble
```

이 분리는 이후에 중요해졌다. 로그인, 설정, 히스토리, SSE 스트리밍이 들어오면서 메시지 종류가 계속 늘어났기 때문이다. 경계를 나누지 않았다면 UI 코드와 VS Code API 코드가 계속 섞였을 것이다.

## 폴더 구조도 다시 정리했다

초기 확장은 `extension/klee-code` 하위에 만들어졌다. VS Code 확장 템플릿을 그대로 가져온 흔적도 있었고, `node_modules`나 빌드 산출물이 커밋에 섞인 적도 있었다.

이후 `extension/` 자체를 확장 패키지 루트로 정리했다. Extension Host와 Webview UI의 위치도 다시 잡았다.

정리 후 기준은 단순했다.

- VS Code API와 직접 닿는 코드는 `extension-host`에 둔다.
- React UI는 `webview-ui`에 둔다.
- 메시지 타입은 protocol 파일에 모은다.
- 빌드 결과와 의존성 디렉터리는 소스 구조에서 제외한다.

이때부터 Klee Code는 "확장 템플릿을 조금 고친 프로젝트"가 아니라, Extension Host와 React Webview를 분리한 구조가 됐다.

## 이 단계에서 바뀐 것

이 시기의 변화는 사용자 입장에서 가장 눈에 잘 보였다. OutputChannel에 찍히던 답변이 채팅 화면으로 들어왔고, 새 대화 버튼이 생겼고, 백엔드 상태도 화면에서 볼 수 있게 됐다.

하지만 더 중요한 변화는 내부 구조였다. UI가 생기면서 자연스럽게 책임이 나뉘었다.

처음에는 command 하나로 충분했다. 하지만 채팅 제품이 되려면 화면 상태, 메시지 프로토콜, 에디터 컨텍스트, 백엔드 호출이 분리되어야 했다. 이 시기에 그 구조를 잡아둔 덕분에 이후 스트리밍과 히스토리 기능을 붙일 수 있었다.

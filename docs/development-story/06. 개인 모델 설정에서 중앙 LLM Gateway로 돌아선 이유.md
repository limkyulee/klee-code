# 개인 모델 설정에서 중앙 LLM Gateway로 돌아선 이유

관련 커밋: `289bb0a LLM Connection Fail Exception`, `aada2e7 Change Model Config User Pref -> On Prem`, `cb5636a fix extension compile error`

로그인과 사용자 설정을 붙이던 시점에는 사용자별 모델 서버 URL을 저장하는 구조가 있었다. 사용자가 본인의 Ollama 주소와 모델을 설정하고, 백엔드가 그 설정을 읽어 채팅을 수행하는 방식이었다.

개인 개발 환경만 생각하면 이 구조는 편하다. 각자 로컬 Ollama를 띄우고, 본인 URL을 저장하면 된다. 하지만 Klee Code의 방향과는 맞지 않았다.

Klee Code는 개인용 LLM 클라이언트가 아니라 온프레미스 AI Coding Assistant다. 회사 내부망에서 승인된 모델만 쓰게 하려면 사용자가 임의의 서버 주소를 입력할 수 있으면 안 된다.

그래서 모델 설정 구조를 다시 바꿨다.

## 사용자별 URL 입력을 제거했다

`aada2e7` 커밋에서 가장 큰 변화는 사용자별 model config 도메인을 제거하고 중앙 `llm` 도메인을 추가한 것이다.

기존에는 `modelconfig` 도메인이 있었다. 사용자별 provider, base URL, model을 저장하는 구조였다. 리팩토링 후에는 이 흐름을 삭제했다.

대신 `LLMGateway`를 만들고, 모델 서버 주소는 `klee.llm.base-url` 설정으로만 관리하게 했다. 사용자가 바꿀 수 있는 값은 `selectedModel`, `temperature`, `responseLanguage` 같은 preference로 제한했다.

구조는 이렇게 바뀌었다.

```text
이전:
UserModelConfig
  -> user provided Ollama URL
  -> user selected model
  -> ChatService creates client per user

이후:
LLMGateway
  -> operator managed base URL
  -> central Ollama server
  -> user preference only within allowed runtime
```

이 변화로 Klee Code의 성격이 더 명확해졌다. 사용자는 모델 서버를 고르는 사람이 아니라, 운영자가 준비한 중앙 모델 환경 안에서 설정만 조정하는 사람이다.

## 모델 목록도 실제 Ollama 기준으로 바꿨다

처음에는 `application.yml`에 모델 목록을 두는 방식도 가능했다. 하지만 실제 운영에서는 YAML에 적힌 목록과 Ollama 서버에 설치된 모델이 어긋날 수 있다.

그래서 `/models`는 중앙 Ollama API를 호출해 실제 설치 모델을 조회하도록 바꿨다. 이 요구사항에는 MCP가 필요하지 않았다. 모델 목록은 Ollama가 이미 API로 제공하고, 백엔드가 그것을 조회하면 된다.

다만 사용자가 임의 서버를 조회하는 것은 아니다. 어디까지나 운영 설정의 중앙 Ollama 서버를 기준으로 한다.

## 연결 실패를 사용자 메시지와 운영 로그로 나눴다

Ollama 연결 실패도 이 시기에 정리했다.

로컬에서 모델 서버가 꺼져 있거나 주소가 틀리면, 내부 예외가 그대로 화면에 노출될 수 있었다. 사용자는 Netty나 connection refused 스택 트레이스를 볼 필요가 없다. 대신 "모델 서버에 접근할 수 없다"는 수준의 메시지를 받아야 한다.

그래서 `ChatModelExceptionMapper`를 추가해 LLM 연결 실패를 사용자 친화적인 메시지로 바꿨다. 내부 로그에는 원인을 남기고, UI에는 의미 있는 오류만 전달한다.

이후 `/models` 호출에서도 비슷한 문제가 있었다. 로컬에서 `ollama list`는 되는데 백엔드 모델 목록은 비어 보이는 상황이었다. 원인은 실행 환경의 네트워크 경계였다.

Docker Compose 안에서는 `http://ollama:11434`가 맞다. 하지만 백엔드를 호스트에서 JAR로 실행하면 `ollama`라는 Docker 내부 DNS 이름을 해석할 수 없다. 이 경우에는 `http://localhost:11434`를 봐야 한다.

그래서 `start.sh` 로컬 실행 경로에서는 `KLEE_LLM_BASE_URL` 기본값을 `localhost`로 잡도록 했다. 반면 Compose 기본 설정은 `ollama`를 유지했다.

이 문제는 단순 설정 오류가 아니라, 로컬 실행과 Docker 실행의 네트워크 모델이 다르다는 것을 분명히 보여준 사례였다.

## 온프렘 구조로 돌아선 이유

사용자별 URL 설정은 기능적으로는 그럴듯하다. 하지만 포트폴리오에서 Klee Code가 말하려는 바와 다르다.

이 프로젝트의 핵심은 "내가 원하는 아무 LLM에 연결하는 확장"이 아니다. "회사 내부망의 중앙 모델 서버를 기준으로 정책을 강제하는 코딩 어시스턴트"다.

그래서 중앙 LLM Gateway 리팩토링은 단순한 코드 정리가 아니었다. 제품의 정체성을 다시 잡은 작업이었다.

이후 README와 설계 문서도 같은 방향으로 바뀌었다. 외부 Claude/GPT 교체 가능성을 강조하던 초기 설명에서, 중앙 Ollama 기반 온프레미스 구조를 강조하는 쪽으로 좁혔다. 기능을 넓히는 대신 제품의 기준을 더 선명하게 만든 셈이다.

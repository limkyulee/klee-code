# SSE 스트리밍과 답변 렌더링을 다듬은 과정

관련 커밋: `c8b6ee2 SSE Calling Process`, `192ee75 fix Chat ui`, `467a050 Dockerfile setting`

React Webview까지 붙이고 나면 다음 문제가 보인다. 사용자는 LLM 응답을 기다리는 동안 빈 화면을 보고 싶어 하지 않는다. 모델이 답을 생성하는 중이라면 토큰이 들어오는 대로 화면에 보여주는 것이 자연스럽다.

그래서 `/chat` 단발 응답에 더해 `/chat/stream`을 추가했다. 백엔드는 Spring MVC의 `SseEmitter`로 `progress`, `token`, `done` 이벤트를 보냈고, extension host는 fetch response body를 읽으면서 SSE 이벤트를 파싱했다.

이때부터 Klee Code는 요청-응답형 도구에서 스트리밍 채팅 도구로 바뀌었다.

## 스트리밍은 단순히 빠르게 보이는 기능이 아니었다

처음 SSE를 붙였을 때는 응답이 실시간으로 들어오는 것 자체가 핵심이라고 생각하기 쉽다. 하지만 실제로 써보면 더 많은 문제가 보인다.

progress 메시지가 너무 자주 바뀌면 화면이 깜빡인다. 토큰이 들어올 때마다 스크롤을 과하게 호출하면 오히려 느려 보인다. 스트림을 파싱하는 과정에서 공백을 잘못 다루면 문장이 붙어 보인다.

실제로 답변이 `Hello!I'manAIassistant...`처럼 붙어 보이는 문제가 있었다. 백엔드 로그에는 앞 공백이 살아 있었는데, 화면에서는 사라졌다. 원인은 모델 응답이 아니라 SSE 파서와 전송 포맷이었다.

LLM 토큰은 앞 공백을 포함하는 경우가 많다. 그런데 `data:` 필드를 파싱하면서 `trimStart()`나 `trim()`을 쓰면 이 공백이 사라진다. 사람에게는 단순 공백이지만 모델 응답에서는 의미 있는 데이터다.

최종적으로는 SSE token payload를 JSON 문자열로 인코딩해 보내는 방식으로 정리했다. 클라이언트는 `JSON.parse`로 원래 문자열을 복원한다. 이렇게 하면 토큰 앞 공백도 데이터로 보존된다.

이 문제를 겪으면서 배운 것은 명확했다. 스트리밍에서는 공백도 프로토콜의 일부로 다뤄야 한다.

## Markdown 응답을 코드 답게 보여주기

AI 코딩 어시스턴트의 답변은 대부분 Markdown이다. 설명과 코드가 섞이고, 코드 펜스가 자주 나온다. 처음에는 텍스트 그대로 보여주는 수준이었지만, 실제 사용하기에는 부족했다.

그래서 Markdown fenced code block을 코드 박스로 렌더링했다. 이후 코드 블록 오른쪽 상단에 복사 버튼을 붙이고, 언어별 syntax highlight도 넣었다.

초기에는 직접 정규식 기반 토큰 색상을 처리하려고 했다. 하지만 이 방식은 유지보수하기 어렵고 언어별 예외가 많다. 결국 `highlight.js` 기반으로 바꿨다.

이 결정은 작지만 중요했다. 직접 구현할 필요가 없는 영역은 검증된 라이브러리를 쓰는 편이 맞다. Klee Code의 핵심은 syntax highlighter를 만드는 것이 아니라, 코딩 어시스턴트의 대화 흐름과 보안 경계를 설계하는 것이다.

## 입력창도 계속 고쳤다

채팅 도구에서 입력창은 계속 쓰는 영역이다. 이 부분도 여러 번 바뀌었다.

입력값이 있을 때만 전송 버튼을 활성화하고, Enter는 전송, Shift+Enter는 줄바꿈으로 정리했다. textarea 이벤트가 기대대로 잡히지 않아 `onKeyDownCapture`로 바꾼 일도 있었다.

전송 중에는 전송 아이콘을 중단 아이콘으로 바꾸고, 클릭하면 현재 스트리밍 요청을 abort하도록 했다. LLM 답변이 길어질 수 있기 때문에 중단은 기본 기능에 가깝다.

이 시기에 입력창에는 모델 표시와 승인 방식 드롭다운 같은 UI도 들어갔다. 다만 승인 방식은 당시 실제 권한 정책과 연결된 기능은 아니었다. 나중에 구현한다면 단순히 메시지 payload에 값을 하나 추가할 문제가 아니라, 세션 단위의 도구 실행 권한이나 파일 접근 정책으로 연결해야 한다고 정리했다.

## 이 단계에서 생긴 기준

SSE를 붙이면서 Klee Code는 더 제품에 가까워졌다. 하지만 이 단계의 핵심은 "스트리밍을 넣었다"가 아니다.

더 정확히는 다음 문제를 해결한 것이다.

- 응답 생성 중 상태를 어떻게 보여줄 것인가.
- 토큰 앞 공백을 어떻게 보존할 것인가.
- Markdown 코드 블록을 어떻게 읽기 좋게 보여줄 것인가.
- 사용자가 응답 생성을 중단할 수 있게 할 것인가.
- 입력창의 키 동작을 일반 채팅 도구처럼 맞출 것인가.

이런 문제들은 하나씩 보면 작다. 하지만 실제 도구의 완성도는 이런 작은 부분에서 결정된다. 이 시기를 지나면서 Klee Code는 "응답을 받아오는 확장"에서 "대화 UI를 가진 코딩 어시스턴트"로 이동했다.

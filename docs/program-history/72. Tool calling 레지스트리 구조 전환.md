# 72. Tool calling 레지스트리 구조 전환

- 상태: 완료
- 작성일: 2026-07-01

## 질의

사용자가 현재 tool calling 흐름은 유지하되, tool 이름과 설명을 여러 곳에 하드코딩하지 말고 서비스 확장에 맞는 registry/schema/executor 구조로 변경해 달라고 요청했다.

## 처리 내용

백엔드 `AgentService`, `ToolPolicyService`, 기존 tool DTO/서비스 구조를 확인했다. Extension Host의 `localTools.ts`가 `toolName` switch로 `read_file`, `search_files`를 직접 분기하는 것도 확인했다.

백엔드에는 `ToolArgumentSchema`, `ToolSchema`, `ToolRegistry`, `ToolPromptService`, `ToolExecutorService`를 추가했다. `AgentService`는 더 이상 tool 목록과 프롬프트 설명 문자열을 직접 소유하지 않고, `ToolPromptService`와 `ToolExecutorService`를 통해 tool calling 루프를 수행하도록 변경했다. `ToolPolicyService`는 자체 Set 대신 `ToolRegistry`를 조회하도록 변경했다.

Extension Host에는 `localToolRegistry.ts`를 추가해 로컬 tool schema와 executor를 한 곳에 등록했다. `localTools.ts`는 registry에서 tool을 조회해 실행하고, 알 수 없는 tool만 실패 응답으로 반환하도록 단순화했다.

관련 설계 문서 `docs/DESIGN.md`, `docs/STRUCTURE_V2.md`에는 tool schema/registry/policy/executor 구조와 Extension Host local executor registry 경계를 반영했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 단순 기능이어도 tool 이름/설명/실행 분기가 여러 파일에 하드코딩되는 구조를 피하고, 서비스 확장 시 유지보수 가능한 tool calling 구조로 바꾸려는 요청이었다.
- 선택한 기술/방식: 백엔드 tool 도메인에 schema record와 registry 서비스를 두고, 프롬프트 렌더링은 `ToolPromptService`, 정책 검증과 결과 대기는 `ToolExecutorService`로 분리했다. Extension Host는 `LocalToolDefinition` 배열 기반 registry로 로컬 실행기를 등록했다.
- 검토한 대안: `AgentService` 내부 문자열과 `ToolPolicyService` Set만 정리하는 방식은 하드코딩 위치만 줄일 뿐 schema/executor 경계가 생기지 않아 제외했다. 백엔드가 실제 파일 read/search를 실행하는 방식은 로컬 워크스페이스 접근 경계와 맞지 않아 제외했다.
- 선택 이유: 기존 SSE tool 요청/결과 흐름은 유지하면서 tool 정의, prompt schema, policy, executor 조율 책임을 분리할 수 있다. 새 tool 추가 시 백엔드는 registry 항목과 정책을 확장하고, Extension Host는 local executor registry 항목을 추가하는 구조가 된다.

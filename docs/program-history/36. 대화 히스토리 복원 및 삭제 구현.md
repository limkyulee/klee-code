# 36. 대화 히스토리 복원 및 삭제 구현

- 상태: 완료
- 작성일: 2026-06-25

## 질의

`conversations` MongoDB 컬렉션 기반으로 대화 목록, 상세 복원, 이어가기, 완전 삭제를 구현하고 기존 `/audit/chat-history` 호환 엔드포인트를 유지해 달라는 요청.

## 처리 내용

백엔드의 기존 `audit` 히스토리 그룹화와 채팅 저장 흐름을 확인한 뒤 `com.kleecode.backend.conversation` 도메인을 `controller`, `service`, `repository`, `dto` 계층으로 추가했다. `ConversationService`가 `conversations` 메타데이터를 관리하고, 기존 `audit_logs`만 있는 대화는 목록 조회 시 lazy backfill하도록 구현했다.

채팅 시작/성공/실패 시 `ChatService`가 `AuditLogService` 저장 결과를 기준으로 conversation 시작/완료 상태를 갱신하게 연결했다. 상세 조회는 conversation 소유권을 먼저 확인한 뒤 `audit_logs`를 시간순으로 읽어 `user`, `assistant`, `error` 메시지로 변환한다. 삭제는 소유권 확인 후 `conversations`, `audit_logs`, Spring AI `ChatMemory.clear(conversationId)`를 함께 수행한다.

확장 host API 클라이언트와 webview 프로토콜에 `/conversations` 목록/상세/삭제 호출, `SELECT_CONVERSATION`, `DELETE_CONVERSATION`, `CONVERSATION_LOADED`, `CONVERSATION_DELETED` 메시지를 추가했다. 웹뷰 히스토리 항목 클릭 시 메시지 목록을 복원하고, 선택된 대화를 활성 표시하며, 휴지통 삭제 버튼과 스트리밍 중 비활성화 처리를 추가했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 사용자가 `audit_logs` 기반 그룹 목록만으로는 히스토리 클릭 복원/이어가기/삭제가 부족하므로 `conversations` 컬렉션을 별도 메타데이터 저장소로 추가하라고 요청했다.
- 선택한 기술/방식: Spring Data MongoDB `ConversationRepository`, `Conversation` 문서 record, `ConversationService` 중심 구현, 기존 `ChatHistoryItem` 호환 응답 재사용, Spring AI `ChatMemory.clear(String)` 직접 호출, extension host의 활성 `conversationId` 교체 방식.
- 검토한 대안: 메시지 본문을 `conversations`에 중복 저장하는 방식은 원본이 `audit_logs`라는 요구와 불일치하고 데이터 정합성 부담이 커서 제외했다. 기존 `AuditLogService`에 계속 그룹화/상세/삭제 책임을 두는 방식은 새 도메인 패키지 요구와 책임 분리가 맞지 않아 제외했다.
- 선택 이유: 메타데이터와 턴 원본을 분리해 기존 데이터 호환성을 유지하고, `/audit/chat-history`는 새 서비스 위임으로 안정적으로 호환된다. 삭제 범위도 한 서비스에 모아 유지보수성과 테스트 가능성을 높였다.

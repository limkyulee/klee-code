# 41. start.sh 확장 빌드 로그 개선

- 상태: 완료
- 작성일: 2026-06-28

## 질의

`start.sh` 실행 시 Ollama URL을 빌드 시점에 지정해야 하는지 확인하고, 백엔드/확장 빌드 상태와 VSCode extension 빌드 오류 로그 확인 방법을 개선해 달라는 요청.

## 처리 내용

`start.sh`, `extension/package.json`, `backend/build.gradle`, `backend/src/main/resources/application.yml`, 기존 로그를 확인했다. Ollama URL은 빌드 필수값이 아니라 실행 후 사용자별 모델 설정으로 저장되는 값이므로 스크립트에서 강제하지 않는 것으로 판단했다.

`start.sh`에 확장 빌드 전 Node/npm 진단, `.node-version` 기반 fnm 적용, npm script Node 버전 검증, `logs/extension-build.log` 기록, 확장 빌드 실패 시 로그 경로 안내를 추가했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 확장 빌드가 멈추거나 실패할 때 터미널에 원인 로그가 명확히 남지 않았고, `/usr/local/bin/node` 20.12.2가 Vite 요구 버전보다 낮아 빌드가 실패했다.
- 선택한 기술/방식: 기존 bash 스크립트 안에서 fnm이 있으면 `.node-version`을 적용하고, `npm ci`/`npm install`/`npm run package` 출력을 `tee`로 `logs/extension-build.log`에 남기도록 구성했다.
- 검토한 대안: Ollama URL을 빌드 환경변수로 강제하는 방식은 사용자별 런타임 설정 구조와 맞지 않아 선택하지 않았다. Node 설치를 스크립트에서 자동 수행하는 방식은 네트워크와 로컬 환경 부작용이 커서 선택하지 않았다.
- 선택 이유: 빌드 실패 원인을 빠르게 확인할 수 있고, 기존 사용자별 모델 설정 구조와 start.sh의 역할을 유지하면서 변경 범위를 스크립트에 한정할 수 있다.

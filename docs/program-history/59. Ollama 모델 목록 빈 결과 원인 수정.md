# 59. Ollama 모델 목록 빈 결과 원인 수정

- 상태: 완료
- 작성일: 2026-06-30

## 질의

로컬에서 `ollama list`는 정상인데 백엔드 `ModelController`의 모델 목록 조회 결과가 비어 있고, `LLMGateway`에 추가한 모델 목록 조회 로그가 `backend.log`에 찍히지 않는 원인을 확인해 달라는 요청.

## 처리 내용

`ModelController`, `LLMGateway`, `OllamaApiConfig`, `application.yml`, `start.sh`, `docker-compose.yml`, `logs/backend.log`, 확장 webview 메시지 처리 코드를 확인했다.

`/models`는 `LLMGateway.availableModels()`를 바로 호출하지만, 현재 로컬 JAR 실행은 `start.sh`가 MongoDB만 Compose로 띄우고 백엔드는 호스트에서 실행한다. 반면 `application.yml`의 기본 Ollama 주소는 Docker Compose 내부 DNS 이름인 `http://ollama:11434`라서 호스트에서 실행되는 백엔드는 `ollama` 호스트명을 해석하지 못한다. 실제 `logs/backend.log`에도 `Failed to resolve 'ollama'` 오류가 남아 있었다.

또한 확장 host의 `postModels()`는 모델 조회 실패를 catch한 뒤 빈 배열 `[]`로 webview에 보내므로, UI에서는 오류 대신 모델 목록이 없는 것처럼 보인다. `LLMGateway`의 기존 `Retrieved ... models` 로그는 `ollamaApi.listModels()`가 성공한 뒤에만 찍히므로 연결 실패 시 찍히지 않는 구조였다.

`start.sh`로 로컬 백엔드를 띄울 때 `KLEE_LLM_BASE_URL`이 없으면 `http://localhost:11434`를 사용하도록 지정했다. 그리고 모델 목록 조회 실패 시 `LLMGateway`가 실제 접근한 Ollama base URL과 예외를 `WARN` 로그로 남기도록 했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 로컬 Ollama는 정상인데 백엔드가 Docker 내부 호스트명 `ollama`를 바라봐 모델 목록 조회와 채팅 호출이 실패했다.
- 선택한 기술/방식: Compose 전체 스택이 아닌 `start.sh` 로컬 JAR 실행 경로에서만 `KLEE_LLM_BASE_URL` 기본값을 `http://localhost:11434`로 설정했다. `LLMGateway`에는 실패 로그를 추가했다.
- 검토한 대안: `application.yml` 기본값 자체를 `localhost`로 바꾸는 방식은 Docker Compose 백엔드 실행 경로와 충돌할 수 있어 제외했다. 확장 UI의 catch 동작까지 바꾸는 방식은 화면 정책 변경 범위가 커서 이번 원인 수정에서는 제외했다.
- 선택 이유: 로컬 실행과 Compose 실행의 네트워크 경계를 분리해 기존 Compose 설정을 유지하면서, 로컬 개발 실행만 실제 환경에 맞게 보정하는 방식이 변경 범위가 작고 재현성이 좋다.

## 결과

로컬 실행 시 백엔드가 Docker 내부 호스트명 대신 호스트의 Ollama 서버를 바라보도록 수정했고, 앞으로 모델 목록 조회 실패 원인이 `backend.log`에 남도록 했다.

# 8. ai_chat_memory 읽기 쓰기 흐름 분석

- 상태: 완료
- 작성일: 2026-06-23

## 질의

`ai_chat_memory`에 데이터를 넣는 경로와 다시 읽어오는 경로가 코드에서 어떻게 연결되는지 분석 요청이 있었다.

## 처리 내용

백엔드와 확장 코드를 함께 확인해 `conversationId` 기반 연속 대화 흐름을 추적했다.

- 확장은 세션 시작 시 `conversationId`를 `randomUUID()`로 생성하고, 새 대화 시 다시 생성한다.
- `buildChatRequest()`가 `conversationId`, 선택 코드, 질문, 주변 컨텍스트를 요청 바디로 만든다.
- 백엔드 `ChatService`는 `ChatClient` 호출 시 `advisors(a -> a.param(CONVERSATION_ID_KEY, request.getConversationId()))`로 같은 키를 전달한다.
- `ChatConfig`는 `MessageChatMemoryAdvisor`를 기본 advisor로 등록하고, `ChatMemory`는 `spring-ai-starter-model-chat-memory-repository-mongodb`의 자동 설정에 맡긴다.
- 설계 문서에서 `ai_chat_memory` 컬렉션이 모델 호출 직전 읽기와 호출 직후 쓰기에 쓰인다고 명시한 부분을 대조해, 별도 저장소가 아니라 Spring AI advisor가 관리하는 연속 대화 메모리임을 정리했다.

## 기능 추가/변경 결정 기록

- 요청 배경: `ai_chat_memory`에 쓰는 코드와 다시 읽는 코드가 어디인지 확인이 필요했다.
- 선택한 기술/방식: Spring AI `MessageChatMemoryAdvisor` + MongoDB 자동 설정 조합으로 `conversationId`를 키 삼아 메모리를 읽고 쓴다.
- 검토한 대안: 애플리케이션에서 `MongoChatMemoryRepository`를 직접 구현해 읽기/쓰기를 명시적으로 제어하는 방식도 가능하지만, 현재 코드는 자동 설정과 advisor에 위임한다.
- 선택 이유: 설정과 추상화가 분리되어 있고, `ChatService`는 메모리 저장 방식과 분리된 채 메시지 생성과 호출에만 집중할 수 있다.

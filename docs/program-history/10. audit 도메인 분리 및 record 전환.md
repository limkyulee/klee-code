# 10. audit 도메인 분리 및 record 전환

- 상태: 완료
- 작성일: 2026-06-23

## 질의

`audit`가 `chat` 하위에 섞여 있으니 별도 도메인으로 분리하고, MVC 기준으로 `controller`, `service`, `repository`, `dto` 구조를 맞춘 뒤 `class` 대신 `record`를 사용해 달라는 요청이었다. 이 규칙을 코드 규칙에도 추가해 달라고 요청했다.

## 처리 내용

- `chat/audit`에 있던 감사 로그 코드를 `com.kleecode.backend.audit`로 분리했다.
- 감사 로그 저장소와 서비스는 각각 `audit/repository`, `audit/service`로 이동했다.
- 감사 로그 문서는 불변 `record`로 바꾸고, 상태 변경은 새 record를 반환하는 방식으로 처리했다.
- `ChatRequest`, `ChatResponse`, `CodeContext`, `SelectionRange`를 `record`로 전환했다.
- `ChatService`와 테스트 코드를 새 record accessor 방식에 맞게 수정했다.
- Spring Boot 4.1 환경에서 `@MockBean`이 없어서 테스트를 `@MockitoBean`으로 변경했다.
- 프로젝트 규칙 파일인 `AGENTS.md`에 도메인 분리, MVC 패키지 구조, record 사용 규칙을 추가했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 감사 로그가 채팅 도메인 안에 섞여 있어 책임 경계가 흐렸고, DTO/문서 객체도 일관된 immutable 스타일로 정리할 필요가 있었다.
- 선택한 기술/방식: 감사 로그를 별도 `audit` 도메인으로 분리하고, 데이터 전달 객체와 감사 로그 문서를 `record`로 모델링했다. 감사 로그 갱신은 mutable 변경 대신 새 record를 만들어 저장하는 방식으로 바꿨다.
- 검토한 대안: 기존처럼 Lombok `class`와 setter 기반으로 유지하는 방식, 또는 record를 쓰되 내부 상태를 강제로 갱신하는 방식은 제외했다.
- 선택 이유: 도메인 경계를 명확히 하고, MVC 패키지 구조를 통일하며, 불변 record를 사용해 데이터 흐름을 단순하게 만드는 쪽이 유지보수성과 일관성이 높다.

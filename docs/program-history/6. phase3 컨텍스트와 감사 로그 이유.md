# 6. Phase 3 컨텍스트와 감사 로그 이유

- 상태: 완료
- 작성일: 2026-06-23

## 질의

`phase3`에서 컨텍스트 수집과 감사 로그 저장을 하는 이유를 설명해 달라는 요청을 받았다.

## 처리 내용

- `DESIGN.md`의 ADR-3, ADR-4, 데이터 흐름, 구현 단계를 확인했다.
- `README.md`의 Phase 3 설명과 설계 결정 요약을 확인했다.
- `backend/src/main/java/com/kleecode/backend/chat/repository/package-info.java`에서 Phase 3에 `AuditLogRepository`가 예정되어 있음을 확인했다.
- 이를 바탕으로 Phase 3의 목적을 두 축으로 정리했다.
  - 컨텍스트 수집: 답변 품질을 높이기 위해 코드/도메인 맥락을 모델 프롬프트에 주입한다.
  - 감사 로그 저장: 사용자, 시각, 모델종류, 외부전송여부, 프롬프트, 응답을 저장해 규제 감사와 추적성을 확보한다.

## 기능 추가/변경 결정 기록

- 요청 배경: Phase 3의 목적이 설계상 왜 필요한지에 대한 설명 요청이 있었다.
- 선택한 기술/방식: 설계 문서와 README에 명시된 Phase 3 정의를 기준으로, 컨텍스트 조회와 감사 로그 저장의 목적을 분리해 설명했다.
- 검토한 대안: 컨텍스트 수집과 감사 로그를 하나의 흐름으로만 설명하는 방식도 가능했지만, 답 품질과 규제 증거의 목적이 달라 분리해서 설명하는 편이 더 명확하다고 판단했다.
- 선택 이유: 설계 문서의 ADR-3과 ADR-4가 각각 답변 품질과 감사 대응을 별도 관심사로 다루고 있어, 이를 그대로 반영하는 설명이 가장 일관적이다.

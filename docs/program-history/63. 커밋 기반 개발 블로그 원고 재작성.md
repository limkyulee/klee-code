# 63. 커밋 기반 개발 블로그 원고 재작성

- 상태: 완료
- 작성일: 2026-06-30

## 질의

사용자는 기존 `program-history` 기록이 없을 수 있으니 커밋 기록을 기반으로, 처음 채팅 UI 없이 agent 대화만 테스트하던 시점부터 Klee Code가 차근히 바뀌어 가는 느낌의 기술 블로그 Markdown 원고로 다시 작성해 달라고 요청했다.

## 처리 내용

- `git log --reverse --date=short --pretty=format:'%h %ad %s'`로 전체 커밋 흐름을 시간순으로 확인했다.
- `git diff-tree --no-commit-id --name-status -r`를 사용해 각 커밋의 실제 변경 파일을 확인했고, `node_modules`와 `dist` 등 생성물은 제외하고 소스 변화 중심으로 해석했다.
- 초기 `README.md`, `DESIGN.md`, Phase 0의 `ChatService`, `extension.ts`, 첫 Webview 구현, React Webview 전환, SSE 스트리밍 코드를 커밋 기준으로 확인했다.
- 기존 기능 묶음 중심 원고를 시간순 커밋 기반 원고로 교체했다.
- 초기 OutputChannel 기반 agent 대화 테스트, Webview 채팅 UI, React 전환, SSE 스트리밍, 코드 컨텍스트/감사 로그, 로그인/히스토리, 중앙 LLM Gateway, 실행 환경 정리 순서로 스토리가 이어지게 재구성했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 이전 원고가 기능별 정리에 가까워, 사용자가 원하는 “처음에는 UI 없이 대화만 검증하다가 점점 제품 형태로 바뀌는 개발 기록” 느낌이 약했다.
- 선택한 기술/방식: `program-history`보다 커밋 히스토리를 우선 근거로 삼고, 각 블로그 글 상단에 관련 커밋 해시와 메시지를 명시했다.
- 검토한 대안: 기존 6개 글을 일부만 보완하는 방식도 가능했지만, 글의 관점 자체가 달라져야 해서 문서 세트를 시간순으로 재작성했다.
- 선택 이유: 포트폴리오용 개발 블로그에서는 최종 구조보다 변화 과정이 중요하므로, 커밋 순서에 맞춘 서사가 더 자연스럽고 실제 개발 흐름을 설명하기 쉽다.

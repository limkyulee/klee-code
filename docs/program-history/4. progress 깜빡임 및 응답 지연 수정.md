# 4. progress 깜빡임 및 응답 지연 수정

- 상태: 완료
- 작성일: 2026-06-23

## 질의

질의 후 `progress` 영역이 계속 깜빡이고, 응답이 매우 느린 이유를 분석해 달라는 요청.

## 처리 내용

- 프론트의 `progress` 표시와 스트리밍 렌더링 흐름을 확인했다.
- `llmService.ts`에서 SSE 이벤트를 처리할 때 `await`로 각 이벤트를 직렬화하고 있어, UI 전송이 느려지면 스트림 처리도 같이 느려지는 병목을 확인했다.
- `webviewMessageHandler.ts`에서 `postMessage`를 매 토큰마다 기다리지 않도록 바꿔, 확장 host 쪽에서 webview 전송이 스트림 수신을 막지 않게 수정했다.
- `MessageList.tsx`의 자동 스크롤을 `requestAnimationFrame`으로 묶어, 토큰이 자주 들어올 때 레이아웃/스크롤 호출이 과도하게 반복되는 현상을 완화했다.
- 백엔드 `ChatController.java`는 `ObjectMapper` 주입에 의존하지 않도록 바꾸고, SSE 이벤트를 raw string으로 전송하게 수정했다. 이로써 테스트 컨텍스트에서 빈 누락으로 실패하던 문제가 사라졌다.

## 기능 추가/변경 결정 기록

- 요청 배경: 스트리밍 응답에서 progress 표시가 눈에 띄게 흔들리고, 답변이 늦게 도착하는 체감 문제를 줄여야 했다.
- 선택한 기술/방식: SSE 파서의 이벤트 처리에서 `await`를 제거하고, VS Code webview 전송도 fire-and-forget으로 바꿨다. 프론트는 자동 스크롤을 프레임 단위로 조정했고, 백엔드는 `ObjectMapper` 없이 SSE 문자열을 직접 전송하도록 단순화했다.
- 검토한 대안: progress 메시지를 누적하지 않거나 cursor 애니메이션을 제거하는 방식도 가능했지만, 우선 병목이 큰 전송/스크롤 경로를 먼저 정리하는 쪽이 효과 대비 변경 범위가 작았다.
- 선택 이유: 응답 지연의 핵심은 렌더링 자체보다 스트림 처리의 역압과 반복 스크롤 호출이었다. 이 경로를 끊는 편이 기존 구조와 맞고, 부작용도 적다.

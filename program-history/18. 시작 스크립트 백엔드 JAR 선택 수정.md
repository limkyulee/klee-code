# 18. 시작 스크립트 백엔드 JAR 선택 수정

- 상태: 완료
- 작성일: 2026-06-23

## 질의

`KLEE_CODE_AI_PROVIDER=ollama ./start.sh` 실행 중 백엔드가 시작 직후 종료되고 `backend-0.0.1-SNAPSHOT-plain.jar`에 기본 Manifest 속성이 없다는 로그가 남는 문제를 확인했다.

## 처리 내용

`start.sh`, `logs/backend.log`, `backend/build/libs` 산출물을 확인했다. Gradle 빌드 결과 실행 가능한 Spring Boot JAR와 `-plain.jar`가 함께 생성되는데, 기존 `java -jar build/libs/*.jar`가 셸 glob 순서상 `-plain.jar`를 먼저 선택해 종료되는 것으로 판단했다. `start.sh`에서 `*-plain.jar`를 제외한 실행 가능한 JAR를 찾아 실행하도록 수정했고, 실행 가능한 JAR가 없으면 명확한 오류를 출력하도록 했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 시작 스크립트가 백엔드 실행 가능 JAR가 아닌 plain JAR를 실행해 서버 시작이 실패했다.
- 선택한 기술/방식: `find build/libs -maxdepth 1 -name '*.jar' ! -name '*-plain.jar' -print -quit`로 실행 대상 JAR를 명시적으로 선택한다.
- 검토한 대안: Gradle 설정에서 plain JAR 생성을 끄는 방식도 가능하지만 빌드 산출물 정책을 바꾸는 범위가 더 크다.
- 선택 이유: 시작 스크립트의 실패 지점만 좁게 수정해 기존 Gradle 산출물 구조를 유지하면서 glob 순서 의존성을 제거할 수 있다.

## 검증

`bash -n start.sh`로 문법 오류가 없음을 확인했다. `./gradlew bootJar -q`는 Gradle 캐시 접근 권한 문제로 최초 실패했지만, 승인된 권한으로 재실행해 성공했다. `KLEE_CODE_AI_PROVIDER=ollama` 백엔드 라이브 smoke test는 샌드박스에서 MongoDB 소켓 연결이 차단되어 실패했고, 권한 상승 재실행은 승인되지 않아 완료하지 못했다.

## 결과

`start.sh`가 더 이상 `backend-0.0.1-SNAPSHOT-plain.jar`를 실행하지 않고, 실행 가능한 Spring Boot JAR를 선택해 백엔드를 시작하도록 수정했다.

# 74. Tool 도메인 주의점 수정

- 상태: 완료
- 작성일: 2026-07-01

## 질의

Tool 도메인 분석에서 확인한 주의점들을 계획대로 순차 수정해 달라는 요청.

## 처리 내용

`ToolPolicyService`의 permission mode 판단을 명시화하고, read-only 도구는 `ASK`, `APPROVE`, `FULL`에서 계속 승인 없이 허용하도록 유지했다. `ToolResultRequest.status`를 `ToolResultStatus` enum으로 전환하고 `/agent/tool-results` 필수 필드 검증과 unreadable JSON 요청의 400 응답 처리를 보강했다. `AgentService`의 tool call 태그 파싱은 단일 태그와 순수 JSON만 허용하도록 강화했고, `toolName` 공백 및 null `arguments`를 거부하도록 했다. `ToolExecutorService`는 dispatcher 실패 시 pending result를 즉시 취소하도록 수정했다. Extension의 `localToolRegistry.ts`는 schema 중복 정보를 제거하고 executor 매핑만 보유하도록 정리했다.

기존 `PromptAssemblyServiceTest`는 `/review` 프롬프트 리소스가 한글화된 상태와 맞지 않아, 현재 리소스 문구 기준으로 assertion을 갱신했다.

## 기능 추가/변경 결정 기록

- 요청 배경: Tool 도메인의 정책, 파싱, 결과 검증, pending cleanup, schema 중복 문제가 유지보수 위험으로 확인되었다.
- 선택한 기술/방식: backend `ToolRegistry`를 schema source로 유지하고, extension은 local executor dispatch만 담당하도록 역할을 분리했다. 결과 상태는 Java enum으로 타입화하고 기존 wire value `SUCCEEDED`/`FAILED`는 유지했다.
- 검토한 대안: `ask` 모드에서 모든 도구를 차단하거나 승인 UI를 추가하는 방안이 있었지만, 현재 read-only workspace 도구 사용자 경험을 유지하기 위해 선택하지 않았다.
- 선택 이유: 기존 동작 호환성을 유지하면서도 향후 write/network 도구 추가 시 정책 확장 지점을 명확히 할 수 있고, 잘못된 tool call/result 입력을 조기에 거부해 안정성이 높아진다.

# 32. 사용자 로그인 모델 설정 세션 처리 구현

- 상태: 완료
- 작성일: 2026-06-24

## 질의

사용자 로그인, 사용자별 모델 서버 주소 입력, 세션 처리를 V2 계획에 따라 구현하고 MongoDB 컬렉션 구조와 설계 문서를 함께 작성해 달라는 요청이 있었다.

## 처리 내용

- 백엔드에 `user`, `auth`, `modelconfig`, `security`, `common` 도메인을 추가했다.
- `users`, `sessions`, `user_model_configs` MongoDB 문서와 repository/service/controller를 추가했다.
- `userId + password` 회원가입/로그인, JWT access token, refresh token 세션 저장, 로그아웃, `/auth/me` API를 구현했다.
- 비밀번호는 BCrypt 단방향 해시로 저장하고, refresh token은 조회 가능한 SHA-256 해시로 저장하도록 분리했다.
- `/chat`, `/chat/stream`, `/chat/status`를 인증 필수로 바꾸고, 사용자별 모델 설정이 없으면 `MODEL_CONFIG_REQUIRED` 오류를 반환하도록 했다.
- `audit_logs`에 `userId` 필드를 추가했다.
- 확장 호스트에 `SecretStorage` 기반 refresh token 보관, access token 메모리 보관, 401 refresh 재시도 흐름을 추가했다.
- Webview에 로그인/회원가입 화면과 Ollama 모델 설정 화면을 추가했다.
- `DESIGN_V2.md`, `TECH_STACK.md`, `STRUCTURE_V2.md`, `mongoDB collection ERD.drawio` 산출물을 작성했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 사용자별 로그인과 모델 서버 설정을 먼저 구현하고, 히스토리 기능의 기반이 되는 사용자 식별/세션 구조를 마련해야 했다.
- 선택한 기술/방식: Spring Security, Spring Security OAuth2 JOSE 기반 JWT, BCryptPasswordEncoder, MongoDB 문서 컬렉션, VS Code SecretStorage를 사용했다.
- 검토한 대안: 서버 세션만 쓰는 방식은 VS Code 확장과 SSE 호출에서 쿠키 관리가 번거로워 제외했다. refresh token을 BCrypt로 저장하는 방식은 인덱스 조회가 불가능해 SHA-256 해시 저장으로 조정했다.
- 선택 이유: JWT access token은 확장 HTTP 호출에 단순하고, refresh token 세션은 로그아웃/만료 관리가 가능하다. MongoDB는 기존 ChatMemory와 감사 로그 저장소와 일관된다.

# 60. Slash Skill 기반 구현

- 상태: 완료
- 작성일: 2026-06-30

## 질의

Klee Code v3 계획에 따라 `/스킬명 질문` 형식으로 내부 스킬과 `.klee/skills` 커스텀 스킬을 사용할 수 있는 기반을 구현해 달라는 요청이었다. `.skill` 폴더는 지원하지 않는 조건이었다.

## 처리 내용

백엔드 `ChatRequest`에 `skillCommand`, `kleeContext`를 추가하고, `prompt` 도메인의 `PromptAssemblyService`가 시스템 프롬프트, 내부 스킬, `.klee` rules/skills/hooks, 코드 컨텍스트, 사용자 질문을 합성하도록 구성했다.

내부 프롬프트 리소스를 `backend/src/main/resources/klee-prompts/` 아래에 추가했고, 기본 내부 스킬로 `review`, `explain`, `fix`, `test`를 제공했다. 알 수 없는 slash skill은 `UNKNOWN_SKILL`로 거절하며, 내부 스킬과 커스텀 스킬이 충돌하면 내부 스킬이 우선되도록 했다.

확장에서는 `ChatWebviewMessageHandler`가 첫 토큰 slash command를 파싱하고, 현재 워크스페이스 루트의 `.klee/rules`, `.klee/skills`, `.klee/hooks` Markdown 파일을 읽어 백엔드 요청에 포함하도록 했다. `.skill` 폴더는 읽지 않는다.

`docs/DESIGN_V3.md`에 slash skill 문법, 내부/커스텀 스킬 위치, 프롬프트 합성 순서, 보안 제한을 문서화했다.

## 기능 추가/변경 결정 기록

- 요청 배경: Klee Code 내부 정의 스킬과 사용자 정의 스킬을 `/스킬명` 방식으로 호출하고, 사용자 커스텀 스킬은 `.klee/skills`만 표준 위치로 사용하려는 요청이었다.
- 선택한 기술/방식: 확장이 slash command와 `.klee` Markdown을 구조화해 보내고, 백엔드 `PromptAssemblyService`가 최종 프롬프트를 합성하는 방식으로 구현했다.
- 검토한 대안: `.skill` 폴더를 별칭으로 지원하는 방식은 사용자가 명시적으로 제거를 요청해 제외했다. 확장에서 최종 프롬프트를 조립하는 방식은 백엔드 감사와 향후 다른 클라이언트 대응이 약해 선택하지 않았다.
- 선택 이유: 백엔드가 최종 조립 책임을 가지면 내부 스킬 우선순위, 알 수 없는 스킬 검증, 시스템 프롬프트 관리가 일관되고 유지보수하기 쉽다.

## 결과

Klee Code v3 slash skill 기반을 구현했다. 사용자는 `/review`, `/explain`, `/fix`, `/test` 내부 스킬을 호출할 수 있고, 프로젝트의 `.klee/skills/foo.md`는 `/foo`로 호출할 수 있다. `.skill` 폴더는 지원하지 않는다.

# 45. 중앙 LLM Gateway 리팩토링

- 상태: 완료
- 작성일: 2026-06-29

## 질의

Klee Code를 개인별 Ollama URL을 입력하는 AI 클라이언트 구조에서 기업 내부망의 중앙 Ollama 서버를 사용하는 온프레미스 AI Coding Assistant 구조로 리팩토링해 달라는 요청.

## 처리 내용

기존 `modelconfig` 도메인, 사용자별 Ollama `ChatClient` 팩토리, `/me/model-config` API를 제거했다. 새 `llm` 도메인에 `LLMGateway`, `LlmProperties`, `GET /models`를 추가하고, 새 `preference` 도메인에 `user_preferences`, `GET/PUT /me/preferences`를 추가했다.

`ChatService`는 더 이상 사용자별 URL이나 provider를 조회하지 않고 중앙 `LLMGateway`의 `ChatClient`를 사용한다. 사용자 preference의 `selectedModel`, `temperature`, `responseLanguage`만 반영하며, 모델 서버 연결 실패 메시지는 내부 URL을 노출하지 않도록 바꿨다.

VS Code 확장은 모델 설정 저장 흐름을 preference 저장 흐름으로 교체했다. Settings 화면에서 Ollama URL 입력과 임의 모델 입력을 제거하고, 허용 모델 선택, temperature, 응답 언어 설정만 남겼다. Docker Compose, `.env.example`, README, 설계/구조/기술스택 문서도 중앙 Ollama Gateway 기준으로 갱신했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 기업 내부 소스코드와 문서가 외부 AI 서비스나 사용자 임의 URL로 전송될 수 없도록 보안 중심 아키텍처로 전환해야 했다.
- 선택한 기술/방식: Spring Bean `LLMGateway`가 `klee.llm` 설정으로 Ollama `ChatClient`를 생성하고, MongoDB `user_preferences`에는 개인 UX 설정만 저장하도록 했다.
- 검토한 대안: 기존 사용자별 `baseUrl` 저장 구조를 유지하면서 URL 검증을 강화하는 방식은 SSRF와 내부 주소 오남용 위험을 근본적으로 없애지 못해 선택하지 않았다.
- 선택 이유: 중앙 Gateway 방식은 운영자가 모델 서버 주소와 허용 모델 목록을 통제할 수 있고, 기존 JWT, Conversation, Audit Log, ChatMemory, SSE 구조에 미치는 영향을 LLM 호출 경계로 제한할 수 있다.

# 39. 대화 메타데이터 설계 설명

- 상태: 완료
- 작성일: 2026-06-25

## 질의

사용자가 lazy backfill의 의미와 적용 이유, `conversations` 컬렉션에 메타데이터만 저장한 이유, 대화 제목 결정 시점을 설명해 달라고 요청했다.

## 처리 내용

`ConversationService`, `ChatService`, `Conversation`, `AuditLog`, 관련 테스트와 기존 히스토리 기록을 확인했다. `conversations`는 목록, 소유권, 상태, 삭제 단위를 관리하는 메타데이터 컬렉션이고, 실제 메시지 원본은 `audit_logs`, LLM 컨텍스트 메모리는 Spring AI `ai_chat_memory`가 담당하는 구조임을 확인했다. 목록 조회 시 기존 `audit_logs`만 가진 대화를 `conversations` 문서로 보강하는 lazy backfill 흐름과 첫 질문 기반 제목 생성 흐름을 정리했다.

## 변경 파일

- `program-history/39. 대화 메타데이터 설계 설명.md`

## 검증

테스트는 실행하지 않았다. 코드 변경 없이 설계 설명을 위한 정적 확인만 수행했다.

## 결과

사용자 질문 3개에 대해 실제 구현 기준으로 설명하고, `conversations`, `audit_logs`, `ai_chat_memory`의 책임 차이를 구분해 답변했다.

# 30. Phase4 온프렘 패키징 진행

- 상태: 일부 완료
- 작성일: 2026-06-23

## 질의

`DESIGN.md`의 Phase 4를 진행해 달라는 요청.

## 처리 내용

`DESIGN.md`, `README.md`, 기존 `docker-compose.yml`, 백엔드 Gradle 설정, `application.yml`, `start.sh`를 확인했다. Phase 4 요구인 온프렘 패키징을 위해 Docker Compose가 MongoDB뿐 아니라 Ollama, Ollama 모델 pull 작업, 백엔드 컨테이너까지 함께 기동하도록 확장했다. 백엔드용 멀티 스테이지 `Dockerfile`과 `.dockerignore`를 추가했고, 기본 Ollama 모델 설정을 설계 문서의 `qwen2.5-coder:3b`와 맞췄다.

기존 `start.sh`는 로컬 개발용 JAR 실행 흐름이므로 compose 전체 스택과 8080 포트가 충돌하지 않도록 MongoDB 서비스만 올리게 조정했다. README와 `docs/demo.md`에는 compose 실행 방법, 상태 확인, GIF/영상 촬영 절차를 정리했다. 실제 VS Code Extension Development Host 화면의 GIF/영상 촬영은 로컬 GUI 작업이 필요해 남겨 두었다.

## 기능 추가/변경 결정 기록

- 요청 배경: Phase 4의 목표가 백엔드, Ollama, DB를 온프렘 환경에서 한 번에 재현 가능한 패키지로 묶는 것이었기 때문이다.
- 선택한 기술/방식: Docker Compose 서비스로 `mongodb`, `ollama`, `ollama-model`, `backend`를 정의하고, 백엔드는 `backend/Dockerfile`에서 Gradle `bootJar`로 이미지화했다. 환경값은 `.env.example`과 compose 환경변수로 외부화했다.
- 검토한 대안: 기존처럼 `start.sh`에서 MongoDB만 compose로 띄우고 백엔드는 로컬 JAR로 실행하는 방식은 데모 재현성이 낮아 Phase 4 목표와 맞지 않았다. compose에서 외부 Anthropic 모드까지 동시에 최적화하는 방식은 온프렘 기본 시연 범위를 넓히므로 이번에는 Ollama 기본값을 우선했다.
- 선택 이유: compose 하나로 보안 경계 안의 실행 단위를 명확히 묶을 수 있고, 기존 Spring AI provider 토글 구조를 유지하면서도 로컬 모델 시연을 재현하기 쉽다. `start.sh`는 로컬 개발 흐름으로 남겨 기존 사용성을 깨지 않도록 했다.

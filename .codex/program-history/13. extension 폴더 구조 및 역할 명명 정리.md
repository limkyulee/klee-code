# 13. extension 폴더 구조 및 역할 명명 정리

- 상태: 완료
- 작성일: 2026-06-23

## 질의

사용자는 `extension/` 현재 폴더 구조를 확인하고, 역할 및 기능별로 정확히 구분 또는 명명할 수 있는지 확인하라고 요청했다. 또한 React 현재 버전에서 권장하는 폴더 구조와 VS Code Webview에서 권장하는 구조를 확인한 뒤 진행하라고 요청했다.

## 처리 내용

- `extension/`의 실제 파일 구조, `package.json`, Vite 설정, Extension Host 코드, React Webview UI 코드를 확인했다.
- React 19.2 공식 문서와 VS Code Webview 공식 문서를 확인해 현재 구조와 비교했다.
- Extension Host와 Webview UI의 런타임 경계를 더 명확히 구분했다.
- React UI는 진입점, 채팅 feature, API bridge, protocol, reducer, component, style 영역으로 분리했다.
- VS Code Webview 쪽은 실제 API 역할에 맞게 `AssistantPanel`을 `AssistantViewProvider`로 정리했다.
- 구조 기준을 문서화하기 위해 `extension/STRUCTURE.md`를 추가했다.

## 기능 추가/변경 결정 기록

- 요청 배경: 사용자가 현재 폴더 구조가 React와 Webview 권장 방식에 맞게 역할별로 구분되고 명명되어 있는지 확인하고 정리하길 원했다.
- 선택한 기술/방식: 기존 Vite 기반 React Webview 구조를 유지하면서 `extension-host`와 `webview-ui` 경계를 명확히 하고, React 내부는 feature 기반 구조로 정리했다.
- 검토한 대안: 전체 구조를 `src/webview` 아래로 합치거나 별도 패키지 workspace로 분리하는 방식도 가능했다. 하지만 현재 규모에서는 이동 범위가 커지고 빌드 설정 변경이 과해진다고 판단했다.
- 선택 이유: 기존 빌드 체계를 유지하면서 역할명과 파일 위치만 바로잡는 방식이 가장 안정적이고, VS Code Webview의 Extension Host/Webview 격리 모델과 React feature 구조를 모두 반영할 수 있었다.

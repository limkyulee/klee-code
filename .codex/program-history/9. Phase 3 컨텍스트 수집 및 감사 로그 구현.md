# 6. Phase 3 컨텍스트 수집 및 감사 로그 구현

- 상태: 일부 완료
- 작성일: 2026-06-23

## 질의

사용자는 Phase 3 계획에 따라 코드 컨텍스트 수집과 감사 로그 저장 기능을 구현해 달라고 요청했고, 이어서 현재 채팅 내용을 `program-history` 기록으로 생성해 달라고 요청했다.

## 처리 내용

- 확장 쪽에 선택 코드와 주변 컨텍스트를 조립하는 순수 함수 `buildChatRequest`를 추가했다.
- 활성 에디터의 선택 영역, 파일 경로, 언어, 선택 범위, 주변 라인 스니펫을 `POST /chat` 및 `POST /chat/stream` 요청에 포함하도록 바꿨다.
- 백엔드 `ChatRequest`에 `CodeContext` / `SelectionRange` DTO를 추가해 컨텍스트 메타데이터를 수신하도록 확장했다.
- 감사 로그 전용 도메인(`AuditLog`, `AuditLogRepository`, `AuditLogService`)을 추가하고, 요청 시작/성공/실패 상태를 MongoDB에 저장하도록 채팅 흐름에 연결했다.
- `spring.ai.model.chat` 설정을 기준으로 `외부전송여부`를 기록하고, 로컬 Ollama는 `false`, 외부 모델은 `true`로 저장되도록 했다.
- MongoDB 인덱스 생성을 위해 `application.yml`에 `auto-index-creation: true`를 추가했다.
- README와 설계 문서의 Phase 3 상태를 완료로 갱신했다.

## 기능 추가/변경 결정 기록

- 요청 배경: Phase 3 목표인 "선택된 코드 + 주변 컨텍스트" 수집과 감사 로그 저장을 실제 코드에 반영해야 했다.
- 선택한 기술/방식: 확장 쪽에서는 `buildChatRequest` 순수 함수를 통해 요청 payload를 생성했고, 백엔드에서는 MongoDB 문서 기반 감사 로그와 별도 서비스로 저장 흐름을 분리했다.
- 검토한 대안: 컨텍스트를 백엔드에서 재구성하는 방식도 가능했지만, VS Code 편집기 정보는 확장 쪽이 더 정확하게 알 수 있으므로 클라이언트에서 수집하는 쪽을 선택했다. 감사 로그를 채팅 서비스 내부에 직접 쓰는 방식도 가능했지만, 실패 격리가 약해져 별도 서비스로 분리했다.
- 선택 이유: 컨텍스트 수집은 편집기 상태에 직접 접근하는 쪽이 단순하고 정확하며, 감사 로그 분리는 응답 경로와 저장 실패를 분리해 안정성을 높인다.

## 검증

- `npm run check-types` - 통과
- `./gradlew test` - 시도했으나 `~/.gradle`에 Gradle wrapper 배포판을 다운로드하려다 샌드박스 권한 문제로 중단됨. 추가 권한 승인이 필요했다.

## 결과

선택 코드와 주변 컨텍스트를 요청 payload에 포함하고, 채팅 요청 시작/성공/실패를 MongoDB 감사 로그로 남기는 구조를 구현했다. 다만 백엔드 테스트는 Gradle wrapper 다운로드 권한 문제로 최종 실행까지는 완료하지 못했다.
